# 分批分析功能 - 避免上下文溢出

## 🎯 问题背景

当收集大量数据时（如 100+ 条），如果一次性将所有数据发送给 LLM 进行分析，会导致：
1. **上下文超限**：超过模型的最大 token 限制
2. **性能下降**：大量数据导致推理变慢
3. **质量降低**：信息过载影响分析质量

## ✨ 解决方案：分批分析 + 并行处理

### 核心思路

```
原始方案（❌ 会超限）:
┌─────────────────────────────┐
│ 100 条数据 → LLM 分析       │  ← 超过上下文限制
└─────────────────────────────┘

分批方案（✅ 优化）:
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 批次1(20条) │  │ 批次2(20条) │  │ 批次3(20条) │  ...
│   ↓ 分析     │  │   ↓ 分析     │  │   ↓ 分析     │
│   摘要A      │  │   摘要B      │  │   摘要C      │
└──────────────┘  └──────────────┘  └──────────────┘
        │                │                │
        └────────────────┴────────────────┘
                         ↓
                  合并所有摘要
                         ↓
                  最终综合分析
```

### 并行处理优化

```python
# 传统串行处理（慢）
批次1 → 分析 → 摘要A
                      ↓
        批次2 → 分析 → 摘要B
                            ↓
                批次3 → 分析 → 摘要C

总耗时 = T1 + T2 + T3

# 并行处理（快）
批次1 → 分析 → 摘要A
    ↓
批次2 → 分析 → 摘要B  ← 同时进行
    ↓
批次3 → 分析 → 摘要C

总耗时 = max(T1, T2, T3)
```

## 🛠️ 实现细节

### 1. 新增状态字段

```python
class OpinionState(BaseModel):
    batch_size: int = 20  # 每批处理的数据量
    batch_analyses: List[str] = []  # 各批次的分析摘要
```

### 2. 核心工具函数

#### `get_data_overview()` - 获取数据概览
```python
输入: 无
输出: 
- 总数据量
- 批次数量
- 批次大小
- 数据来源分布

示例输出:
数据概览：
- 总数据量: 100 条
- 批次大小: 20 条/批
- 总批次数: 5 批

建议流程: 分析批次 0-4
```

#### `get_batch_data(batch_index)` - 获取批次数据
```python
输入: batch_index (0, 1, 2...)
输出: 该批次的详细数据（标题、摘要、来源等）

示例:
批次 1 数据（1-20/100）：
1. **新能源车销量创新高**
   来源: 新华网
   摘要: ...
2. ...
```

#### `save_batch_analysis(batch_index, summary)` - 保存批次摘要
```python
输入: 
- batch_index: 批次索引
- summary: 该批次的分析摘要

作用:
- 保存到 state.batch_analyses
- 更新前端进度
- 实时显示分析进展

示例:
批次 0 分析:
- 关键词: 新能源、销量、政策
- 情感: 正面占 60%
- 主要观点: 市场看好...
```

#### `merge_batch_analyses()` - 合并批次摘要
```python
输入: 无（从 state 读取）
输出: 所有批次的综合摘要

示例输出:
批次分析汇总（共 5 批）：

批次 1:
- 关键词: 新能源、销量、政策
- 情感: 正面 60%

批次 2:
- 关键词: 技术创新、充电
- 情感: 正面 55%
...

请基于以上批次分析，提取整体的关键词、情感倾向、热度趋势等。
```

### 3. 分析流程

```
阶段 2: 数据分析（新流程）

1. start_analysis() → 进入分析模式
2. get_data_overview() → 获取概览
   返回：共 100 条数据，分 5 批

3. 对每个批次（可并行）:
   ┌─ 批次 0:
   │    get_batch_data(0) → 获取数据
   │    LLM 分析 → 提取关键词、情感
   │    save_batch_analysis(0, "摘要...") → 保存
   │
   ├─ 批次 1:
   │    get_batch_data(1) → 获取数据
   │    LLM 分析 → 提取关键词、情感
   │    save_batch_analysis(1, "摘要...") → 保存
   │
   └─ ...

4. merge_batch_analyses() → 获取综合摘要

5. 基于综合摘要进行最终分析:
   - 提取整体关键词
   - 计算综合情感得分
   - 生成热度趋势

6. save_analysis_result(...) → 保存最终结果
7. finish_analysis() → 完成
```

## 📊 配置说明

### 前端配置

在左侧面板可以配置：

```
┌─────────────────────────────┐
│ Max Results to Collect: 100 │  ← 总共收集多少条
├─────────────────────────────┤
│ 分批分析大小: 20            │  ← 每批处理多少条
```

**建议配置**:
- 测试环境：`max_results=5`, `batch_size=20`（不会触发分批）
- 小规模：`max_results=20`, `batch_size=20`（1批）
- 中规模：`max_results=50`, `batch_size=20`（3批）
- 大规模：`max_results=100`, `batch_size=20`（5批）

### 后端默认值

```python
OpinionState(
    max_results=100,  # 默认收集 100 条
    batch_size=20,    # 默认每批 20 条
)
```

## 💡 性能优化

### 1. 批次大小调优

| 批次大小 | 优点 | 缺点 | 适用场景 |
|---------|------|------|----------|
| 10条 | 上下文小，处理快 | 批次多，总时间长 | 数据质量差，需精细分析 |
| 20条 | 平衡 | - | **推荐默认值** ✅ |
| 30条 | 批次少 | 上下文大，可能超限 | 数据简洁，模型能力强 |
| 50条 | 批次最少 | 上下文很大，风险高 | 仅限摘要简短的数据 |

### 2. 并行处理

LLM 本身不支持真正并行，但可以：
1. **异步请求**：使用 `asyncio` 同时发起多个批次请求
2. **流式返回**：边分析边保存，提升用户体验
3. **批次缓存**：已分析的批次可缓存，避免重复

### 3. 内存优化

```python
# 只传递必要信息，不是完整数据
get_batch_data(0)  # 只返回该批次
                   # 而不是全部 100 条
```

## ✅ 总结

分批分析功能通过以下方式解决了上下文溢出问题：

1. **分而治之**：将大数据集拆分为小批次
2. **逐批分析**：每批独立分析，避免超限
3. **合并摘要**：基于各批次摘要进行最终分析
4. **并行优化**：支持异步并行处理，提升效率
5. **实时反馈**：用户可见每个批次的进度

**用户无需关心内部实现，系统自动处理分批逻辑！** ✨
